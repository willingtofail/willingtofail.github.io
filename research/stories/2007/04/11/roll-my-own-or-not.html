Should I roll my own?

I want to move my weblog to a system I can work on directly.

Of course I have two choices: I can use a weblog system someone else wrote, like Typo or WordPress, or I can roll my own weblog system.

Who does that anymore?

I don't know many webloggers that use their own blogging system. Most people have transitioned to something like MovableType or WordPress or Typo.

But I'm really picky. That's why I've been uing Conversant for so many years - it really does do weblogging the way I like it. But unfortunately it's not on a platform I can do much real customization with.

So if I want to be able to customize the site, does that eliminate certain options?

I would never consider WordPress, because it's written in PHP, and I hate PHP. Nor would I choose MT because I don't like Perl.

I'm not a big fan of Rails, so I probably won't choose Typo, but I won't rule it out yet.

What must I have?

Be able to post using MarsEdit.

That's no problem, I just have to support the MetaWeblog API or any other API MarsEdit supports.

Be compatible with my old posts.

Nearly 8 years of posts is a lot of data to port over. Either the system I write is compatible with that data entry format, or I write a one-off translator to whatever new system I decide to use.

Be able to display and accept comments.

Conversant is a threaded discussion board (like a usenet group), and I have well over 7000 messages in my site, including my posts.

I don't think I need to maintain threaded comments though.

Have daily, monthly and per-item archive pages.

No brainer.

Ping weblogs.com on updates

No brainer.

RSS/ATOM output.

No brainer.

Tags

I haven't been maintaining categories on my posts for years, but for a long time I've planned to eventually go back through all my posts and tag them. That'll be 'fun'.

Search

No brainer.

What don't I need?

Group blogging

I don't see my weblog ever becoming a group blog.

Users registering and logging in

No use case for it.

Trackback

Never used it, never understood why I'd want it.

What could I use to implement my weblog?

I have a great hosting package with site5.com. It gives me 6 full, completely independent hosting accounts. I'm using one for this site, one for www.roepcke.com, and Steve is using a site for one of his projects.

They support PHP, Perl, Rails, and supposedly it's possible to run Django on it with some work so Python is obviously available.

I'd probably use Python because it's the language I've used the most in the last few years. If I use Python I could use SimpleTAL or Zope3's TAL packages. But apparently site5 doesn't support mod_python.

Of course they support MySQL. They might support other RDMBS but I haven't checked.

Am I nuts?

As you can probably tell, I'm leaning towards rolling my own.

I think it's a healthy thing to do, because I need to get back into a designing and coding and improving frame of mind.

Assuming I roll my own...

I'll need to design the system.

For a while now I've been thinking about the benefits of static rendering. Back when I worked for CEISS I maintained their web site using Frontier, but it was actually an ASP/VBScript site as far as anyone else was concerned. The templates in Frontier contained ASP/VBScript code. When pages were requested the web server would process the ASP/VBScript code. I've long felt this is a great way to maintain dynamic / data-driven web sites.

Could a weblog system work like this, in part or in full?

The thing is, you still want the (post/comment) data in a database so you can index and search it. Right? Or is it reasonable to put the content in files and use a non-RDBMS-based indexing system like Lucene or one of its derivatives to provide the searching capabilities?

Maybe it's a moot question. There's no reason I couldn't maintain whatever type of templates I write using Frontier, but depending on the architecture there might not be any point.

The problem with having Frontier or any system spray semi-static/semi-dynamic files all over the place is those files have to be maintained.

I'm talking myself out of it you see. It's probably best to stick with the current, more MVC-oriented approach. I suppose the system could use static files as a caching system.
